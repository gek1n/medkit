# План: мультисемейность + биллинг + связанные задачи

> Документ создан в рамках длинной дизайн-сессии (обсуждение, без кода) поверх уже
> реализованной сегодня унификации ролей (Локальний/Автономний), обмена фото между
> пирами и лимитов на приглашения. Все решения ниже — согласованы с пользователем.
> Если что-то не помечено как "решено" — это открытый вопрос, не додумывать.

## Контекст: два репозитория

- **Клиент** (`medkit`, Flutter/Dart) — этот репозиторий. Полностью в git,
  ветка `claude/family-group-sync` (или её потомок) уже содержит унификацию ролей,
  обмен фото/документами между пирами, проверки лимитов при приглашении.
- **Сервер** (`medkit-backend`, чистый PHP, без фреймворка/Composer) — лежит рядом
  на Desktop (`C:\Users\user\Desktop\medkit-backend`), **НЕ в git**, деплоится
  вручную через cPanel File Manager (см. `DEPLOY.md`). Структура:
  `medkit_private/src/{Ai,Auth,Firebase,Http,Modules,Security}`, `migrations/*.sql`
  (применяются вручную через phpMyAdmin), `public/index.php` (роутинг + DI без
  контейнера), `cron/cleanup.php`.
- **Важнейшее архитектурное свойство сервера**: он **сознательно ничего не знает**
  о семьях/участниках/лимитах — только opaque `channel_id` + зашифрованные блобы
  (`ciphertext`). Вся логика "кто в какой семье", "сколько слотов" живёт **только
  на клиенте**. Это нужно СОХРАНИТЬ и для биллинга — сервер получает только "у
  аккаунта X активна подписка Y", ничего сверх этого.

## 1. Мультисемейность

**Решение**: простой вариант — человек может состоять в нескольких семейных
группах одновременно (быть "контролёром" у нескольких родителей/родственников).
Никакой миграции на список familyId не делаем.

### Найденный баг (чинить в любом случае, не только ради мультисемейности)

`Members.familyId` сейчас используется для ДВУХ разных смыслов:
- "семья, которую Я создал и за которую плачу" (`FamilyGroupService.createInvite()`
  берёт `owner.familyId`, создаёт если пусто).
- "семья, в которую МЕНЯ пригласили" — `acceptInvite()` и онбординг-конверсия
  (`join_family_screen.dart`) **перезаписывают** это же поле чужим familyId.

Если я плачу за свою семью (familyId=X) и потом принимаю чьё-то приглашение
(familyId=Y) — код молча заменяет мой familyId на Y, ломая мою собственную семью
(следующее моё приглашение пойдёт под чужим Y).

**Фикс**:
- `Members.familyId` означает ТОЛЬКО "семья, которую я создал/оплачиваю". Никогда
  не трогать это поле при `acceptInvite()` или при создании профиля в
  `join_family_screen.dart`.
- Убрать в `acceptInvite()` проверку `owner.familyId != preview.familyId →
  throw GroupJoinException`.
- "В какой семье я гость" отслеживается ИСКЛЮЧИТЕЛЬНО через
  `FamilyPeers.familyId` (уже есть, по одной строке на пира) — этого достаточно.

### `leaveGroup()` — баг усугубится

Сейчас `FamilyPeerSyncService.leaveGroup()` удаляет **всех** пиров без разбора
семьи. С мультисемейностью это станет заметно (выход из одной семьи выкинет из
всех). **Фикс**: `leaveGroup(String familyId)` — принимать параметр, удалять
только пиров этой конкретной семьи (`FamilyPeers.familyId == familyId`).

### UI

- Раздел "Сімейна група" на `family_screen.dart` должен группировать пиров по
  `familyId` (визуально отдельные секции — "Сім'я мами", "Сім'я тестя" и т.д.),
  у каждой секции — своя кнопка "Покинути", вызывающая `leaveGroup(thisFamilyId)`.

## 2. Направление связи (кто кого пригласил)

**Проблема**: `FamilyPeers` сейчас не хранит, кто инициатор связи. Это нужно для:
- Корректного подсчёта слотов — сейчас `FamilyPeersRepository.allPeers().length`
  считает ВСЕХ пиров, включая тех, кто позвал МЕНЯ (баг: входить в чужую семью
  бесплатно, но он же незаконно тратит МОЙ лимит).
- Определения, кто может подарить мне плюшки Family (только тот, кто ПРИГЛАСИЛ
  меня, не наоборот).

**Решение**: добавить колонку `FamilyPeers.invitedMe BOOL` (миграция схемы Drift,
следующий номер schemaVersion после 21). При заполнении:
- Органическое приглашение — тот, кто СКАНИРУЕТ код (`acceptInvite()`),
  ставит `invitedMe = true` для новой записи; тот, кто код СОЗДАЛ
  (`refreshPeers()` на его стороне), ставит `invitedMe = false`.
- Конверсия (онбординг) — новый профиль ставит `invitedMe = true` для инвайтера
  (`join_family_screen.dart`); инвайтер при `refreshPeers()` — `invitedMe = false`.
- Пиры, появившиеся через автопредставление (см. раздел 4) — `invitedMe = false`
  с ОБЕИХ сторон (знакомство через платящего не даёт прав дарить друг другу
  плюшки — дарит только прямой инвайтер).

Существующие (уже засинканные) строки `FamilyPeers` при миграции — направление
неизвестно, ставить `invitedMe = false` по умолчанию (безопасный вариант: не
посчитает случайно чужой слот как свой, просто не даст возможного подарка,
пока связь не пересоздастся).

Подсчёт слотов теперь: `FamilyPeers.where((p) => !p.invitedMe).length` вместо
`allPeers().length`.

## 3. Автопредставление + ленивое (по требованию) создание каналов

**Проблема, которую нашли**: сейчас "семья" технически — звезда через платящего.
Два разных приглашённых участника НЕ видят друг друга вообще (ни имени, ни
данных), пока не отсканируют код друг у друга лично.

**Решение (согласовано)**:
- Когда кто-то присоединяется к семье, платящий автоматически "знакомит" его со
  ВСЕМИ существующими участниками ЭТОЙ семьи — рассылка **только визиток**
  (имя/аватар/personUuid), без доступа к данным.
- Новая таблица (условно `KnownFamilyMembers`: `personUuid`, `name`,
  `avatarIndex`, `familyId`) — визитки людей, которых я ЗНАЮ (вижу в списке
  "Видимість для сім'ї"), но с которыми у меня ЕЩЁ НЕТ настоящего
  зашифрованного канала. Отдельно от `FamilyPeers` (там только реально
  установленные каналы).
- Экран "Видимість для сім'ї" показывает ОБА источника (`FamilyPeers` +
  `KnownFamilyMembers`) как переключатели.
- **Канал создаётся лениво**, только когда реально нужен: пользователь включает
  хоть один тумблер для человека из `KnownFamilyMembers` → устройство отправляет
  платящему (по уже существующему каналу с ним) запрос
  `{'type': 'request_introduction', 'targetPersonUuid': ...}` → платящий (на
  следующей синхронизации) генерирует свежий `channelId` + ключ для этой ПАРЫ и
  пересылает обоим по их существующим каналам с ним (`{'type': 'introduction',
  'peerPersonUuid':..., 'channelId':..., 'key':...}`) → обе стороны создают
  настоящую `FamilyPeers`-запись, тумблер начинает реально работать.
- До готовности канала — UI показывает тумблер в состоянии "Очікуємо з'єднання"
  (тот же паттерн, что уже есть для pending-конверсии и pending-запроса фото).
- **Серверу для этого ничего менять не нужно** — все сообщения (`request_
  introduction`/`introduction`) это такие же generic зашифрованные entity,
  которые уже проходят через существующие `/family-sync/push`/`pull` или
  `/relay/*` — сервер их не отличает от любых других.

### Модель шифрования — явное решение, НЕ единый ключ на семью

Обсуждали и **отклонили** идею одного общего ключа шифрования на всю семью.
Причины (см. полную аргументацию в истории чата, кратко):
1. Взлом ЛЮБОГО ОДНОГО участника при общем ключе компрометирует ВСЮ переписку
   всей семьи, а не только то, чем этот участник лично делился.
2. Ломает саму суть deny-by-default: если у Боба и Кэрол один ключ, Кэрол
   технически может прочитать то, что предназначено только Бобу.
3. Исключение участника при общем ключе требует смены ключа у ВСЕХ разом —
   нетривиальная распределённая задача (именно на этом Signal/WhatsApp тратят
   массу усилий для групп).

**Остаётся попарное шифрование** (уже реализовано), расширенное ленивым
знакомством через платящего (выше). При максимум 8 участниках на план Family
(28 возможных пар) — пересылка через платящего абсолютно посильна, не нужна
настоящая групповая криптография.

## 4. Биллинг — согласованная модель

### Кто что может

- **Приглашать** новых людей в свою семью может только **платящий Family**, в
  рамках своего лимита слотов (`maxAutonomousMembers`).
- **Принять** приглашение может кто угодно, на любом тарифе, бесплатно.
- Приглашённый получает **ВСЕ плюшки Family** (безлимит локальных, синк,
  безлимит сканов/голоса) **кроме права приглашать в эту же семью**.
- Если у приглашённого уже есть свой Plus — обе вещи существуют параллельно:
  свой Plus продолжает оплачиваться и работать, плюшки Family — сверху. Отмена
  своего Plus ничего не ограничивает, пока гость состоит в чужой Family.
- Эффективный тариф = **max(свой реально оплаченный тариф, лучший из активных
  "подарков" от семей, где меня пригласили)**. Считать **динамически** каждый
  раз, никогда не кешировать булевым флагом статично — если распадётся ОДНА из
  двух семей (я гость в двух), доступ от ВТОРОЙ должен остаться рабочим.

### "Превышение лимита" — переиспользуем существующий паттерн

Уже есть в коде (`isMemberBlockedByPlan`/`EllyDeniedScreen` для локальных
профилей сверх лимита): **просматривать и удалять можно всегда, редактировать/
создавать новое — нельзя**, пока не восстановлен доступ. **Никогда не удаляем
данные автоматически** из-за биллинга. Это же правило применяется и к:
- Локальным профилям, заведённым "в долг" по чужой Family-подписке, если та
  подписка позже пропадёт.
- Общему случаю "гостя исключили или семья распалась".

### Сигнал "платит ли ещё мой инвайтер" — без нового сервер-эндпоинта

Устройство гостя никогда не спрашивает сервер о ЧУЖОМ биллинге (утечка
приватности). Вместо этого: устройство ПЛАТЯЩЕГО само проверяет свой статус
(через `/subscription/status`, см. раздел 6) и добавляет поле в уже
существующий периодический `grants_summary`-подобный пуш каждому пиру, например
`payerPlanActive: bool`. Гость пересчитывает свой эффективный план при каждой
синхронизации с этим пиром.

### Грейс-период и распад (только для ПЛАТЯЩЕГО)

1. Неудачная оплата → **5 дней грейс-периода**. Push-напоминание + поп-ап с
   Элли при каждом входе в приложение, "залишилось N днів". **Всё работает без
   ограничений весь грейс-период**, для платящего и для всех гостей.
2. По истечении 5 дней, если не оплачено — **сразу** (без второго ожидания)
   разрыв ВСЕХ связей этой семьи: `leaveGroup(familyId)` со стороны платящего,
   удаляются только общие данные видимости (SharedEntities/FamilyPeers кеш)
   между людьми — **никогда** не удаляются собственные локальные данные
   участников, даже заведённые "в долг" (см. правило выше — freeze, не delete).
3. **Добровольная смена тарифа** (не неудачная оплата, а осознанное действие
   платящего) — показать явное предупреждение ДО смены ("зв'язки з родиною
   будуть розірвані, ви перестанете бачити одне одного", дизайн в стиле
   приложения с иллюстрацией Elly). Подтвердил → разрыв **мгновенный**, без
   грейс-периода (уже предупреждён).
4. **Пуши/поп-апы получает только платящий.** Гости не получают никаких
   проактивных уведомлений о чужом биллинге — узнают реактивно, при следующей
   своей синхронизации.
5. **Модалка "почему изменился доступ"** — показывается **один раз** при первом
   входе после изменения (и у платящего при смене тарифа, и у гостя после
   разрыва/заморозки) — Элли, объяснение причины, кнопка "Змінити план" и
   кнопка "ОК" (просто закрыть). Важно: должна триггериться не только при
   холодном запуске, но и если пользователь уже был открыт в приложении в
   момент изменения (не только на следующий cold start).
6. **Исключение ОДНОГО участника** ("Прибрати") платящим — отдельное, всегда
   доступное действие, не завязанное на биллинг-цепочку. Но **последствия для
   исключённого — те же самые**, что при полном распаде семьи (freeze его
   локальных данных, если они были "в долг").
7. Если после заморозки пользователь позже присоединится к ДРУГОЙ платящей
   Family — разблокировка автоматическая (следует из "считать динамически").

### UI, которого не хватает (весь новый)

- Счётчик занятых слотов у платящего ("5 з 8") в разделе "Сімейна група".
- Небольшой бейдж-корона/адмін у платящего (**только у него**, гостям никаких
  меток/бейджей не добавляем).
- Экран/модалка предупреждения перед добровольной сменой тарифа.
- Модалка "что изменилось и почему" (см. пункт 5 выше).
- Поп-ап Elly + текст push-уведомления для грейс-периода.
- Полные слоты + попытка присоединиться → **generic ошибка** "код недійсний або
  прострочений" (специально НЕ говорим "у них нет мест" — не палим информацию
  постороннему со случайным кодом).
- Непринятые (сгенерированные, но не отсканированные) приглашения НЕ считаются
  занятым слотом — слот тратит только реально присоединившийся участник.
  Отдельная инвалидация при распаде, скорее всего, не нужна: коды и так живут
  всего 30 минут (`PairingController::TTL_MINUTES`), к моменту любой биллинг-
  драмы (дни) они уже неактуальны сами по себе — **проверить при реализации**,
  но, вероятно, ничего доделывать не придётся.

## 5. Сервер (`medkit-backend`) — что нужно добавить

Стиль — как везде в этом бэкенде: чистый PHP без Composer, `Database`
(mysqli-обёртка с prepared statements), `Router`/`Request`/`Response`,
rate-limiting через существующий `RateLimiter`, миграции — сырые пронумерованные
`.sql` файлы, применяемые вручную через phpMyAdmin.

### Новая миграция `migrations/005_billing.sql`

```sql
CREATE TABLE subscriptions (
    account_id CHAR(36) NOT NULL,
    platform VARCHAR(10) NOT NULL,        -- 'ios' | 'android'
    product_id VARCHAR(100) NOT NULL,
    status VARCHAR(20) NOT NULL,          -- 'active' | 'grace' | 'expired' | 'cancelled'
    original_transaction_id VARCHAR(200) NOT NULL,
    expires_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (account_id),
    UNIQUE INDEX idx_transaction (original_transaction_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

(одна активная подписка на аккаунт — достаточно для v1, мульти-подписок не
нужно).

### Новый модуль `src/Modules/Billing/`

- **`SubscriptionController.php`**:
  - `verify(Request)` → `POST /subscription/verify` — клиент шлёт `account_id`,
    `recovery_key_hash` (доказывает владение аккаунтом, тот же паттерн, что
    `AccountController::delete()`), `platform`, `product_id`,
    `receipt`/`purchase_token`. Сервер проверяет чек напрямую у Apple/Google,
    сохраняет/обновляет строку `subscriptions`, возвращает текущий статус.
  - `status(Request)` → `POST /subscription/status` — дешёвый lookup из БД
    (без похода к Apple/Google), для частых проверок при каждом открытии
    приложения.
- **`AppleReceiptValidator.php`** / **`GoogleReceiptValidator.php`** — тонкие
  HTTP-клиенты к App Store Server API / Google Play Developer API, тем же
  стилем, что `Auth/JwtVerifier.php` (сырой curl/openssl, без зависимостей).
- Позже (v2, не блокирует v1): **`WebhookController.php`** — приём
  App Store Server Notifications V2 / Google RTDN для проактивного обновления
  `subscriptions` без ожидания захода пользователя в приложение. Не обязательно
  для рабочего v1 — polling при каждом открытии (см. ниже) достаточен для
  старта.
- Новые роуты в `public/index.php`:
  ```php
  $router->post('/subscription/verify', [$subscriptionController, 'verify']);
  $router->post('/subscription/status', [$subscriptionController, 'status']);
  ```
- Новые `.env`-ключи (заполнение — админская часть, пользователь делает позже
  сам): ключи App Store Server API (issuer ID, key ID, .p8-ключ) или
  `APPLE_SHARED_SECRET` для legacy verifyReceipt; сервис-аккаунт Google Play
  Developer API (возможно, можно переиспользовать существующий Firebase
  service account, если дать ему нужные права — уточнить при реализации).

### Ограничение, которое нужно сохранить

Сервер **не получает** никакого понятия о "семьях"/"слотах"/"участниках" — это
осталось бы полностью на клиенте, как и сейчас. Единственная новая
ответственность сервера — "активна ли подписка аккаунта X, до какого числа".

### Открытый вопрос (не решён, нужно решить перед реализацией)

Нужен ли `account_id` ДО покупки — то есть создаётся ли sync-аккаунт
автоматически в момент покупки Plus/Family (обе доли подразумевают
`serverSync: true`), если пользователь до этого был в чистом локальном режиме?
Рекомендация: да, автоматически создавать при покупке, если ещё нет — но нужно
подтверждение перед реализацией.

## 6. Клиент (`medkit`) — что нужно добавить

- **`planProvider`** перестаёт быть голым `StateProvider<AppPlan>` — становится
  реальным: кеш в SharedPreferences (статус + `expires_at`), обновляется через
  `/subscription/status` при каждом открытии/resume приложения (не отдельным
  polling-циклом, а довеском к уже существующим синкам — не нагружает сервер
  лишний раз). Эффективный план = `max(свой кеш, лучший активный подарок от
  FamilyPeers.invitedMe==true с payerPlanActive==true)`.
- **`SubscriptionService`** (новый) — обёртка над `in_app_purchase`: запуск
  покупки, обработка результата, вызов `/subscription/verify`, "Відновити
  покупки".
- **`PlansScreen`** — подключить кнопки "Купити" к реальному потоку (сначала
  проверить, что там сейчас — декоративная заглушка или что-то уже есть, не
  предполагать).
- Модель данных (Drift, миграция после текущей schemaVersion 21):
  - `FamilyPeers.invitedMe BOOL` (раздел 2).
  - Новая таблица `KnownFamilyMembers` (раздел 3).
  - Логика: перестать перезаписывать `Members.familyId` при accept/конверсии

- `FamilyPeerSyncService`:
  - `leaveGroup(String familyId)` вместо без параметров (раздел 1).

## 7. Рекомендуемая последовательность реализации

1. **Мультисемейность-фикс** (раздел 1) — самостоятельная задача, не зависит от
   биллинга, чинит реальный баг уже сегодня.
2. **Направление в FamilyPeers + корректный подсчёт слотов** (раздел 2) —
   тоже независимо, чинит другой реальный баг (входящие приглашения незаконно
   тратят свой лимит).
3. **Автопредставление + ленивые каналы** (раздел 3) — опирается на 1 и 2
   концептуально, но не блокируется биллингом.
4. **Сервер: таблица подписок + `/subscription/verify`+`/status` + клиентская
   интеграция `in_app_purchase`** (разделы 5-6) — можно строить и тестировать
   против sandbox Apple/Google ещё до того, как готова админская часть
   (реальные продукты в App Store Connect/Google Play Console).
5. **Реальный `planProvider` + эффективный план + `payerPlanActive` в
   grants_summary** (раздел 6) — зависит от пункта 4.
6. **Грейс-период/распад + все новые экраны** (раздел 4 полностью) — зависит
   от пункта 5.

## 8. Явно принятые компромиссы (чтобы новая сессия не сомневалась)

- Шифрование — только попарное, никогда общий ключ на семью (раздел 3).
- Лимиты на слоты — мягкие, только клиентские (сервер физически не может их
  проверять, у него нет понятия "семья") — согласуется с уже существующим
  прецедентом (`maxLocalMembers`/`maxAutonomousMembers` и сегодня чисто
  клиентские).
- Роль сервера в биллинге узкая: "активна ли подписка аккаунта X" — ничего
  больше.
- Никогда не удаляем данные из-за биллинга — только замораживаем
  (view+delete, без edit/create).
- Гости никогда не получают проактивных уведомлений о чужом биллинге.

## 9. Явно НЕ покрыто этим планом

- Реальная настройка продуктов в App Store Connect / Google Play Console —
  админская часть, пользователь делает сам позже.
- Webhook-обработка (App Store Server Notifications V2 / Google RTDN) — v2,
  не блокирует первую рабочую версию.
- Подтверждение: создаётся ли sync-аккаунт автоматически при покупке (раздел 5,
  открытый вопрос).
- Текущее состояние `PlansScreen` не проверялось в рамках этой сессии — нужно
  посмотреть перед началом реализации, а не предполагать.
